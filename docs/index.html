Two Independent Features

CCS Code is not a single tool — it is two tools sharing the same CLI. You can use Migration Intelligence without ever touching the vault, and vice versa. Each has its own commands, its own pipeline, and its own output folder.

⚡ Feature 1 — Migration Intelligence

Point it at a legacy GitHub repo. CCS scans it with a three-tier engine — free AST parsing first, then Haiku for structure discovery, then Sonnet for behavior extraction — and produces a verified migration contract your AI coding agents can implement without guessing.

  • Multi-language AST parser (TS, Python, Java, C#, Go, C++, VB6…)
  • Every claim traced to an exact file + line on GitHub
  • Independent verification pass before agents touch code
  • Interactive dashboard + vis.js system graph auto-generated
  • Agent-ready: Claude Code slash commands + Codex AGENTS.md
  • Enforced human-sign-off gates for low-confidence components

🧠 Feature 2 — Knowledge Vault

Ingest GitHub repos, Claude sessions, Cursor logs, VS Code Copilot histories, and local files. CCS enriches every page with AI summaries and [[wikilinks]], then renders your entire knowledge base as an interactive graph you can browse and query in natural language.

  • Auto-harvest from Cursor, VS Code Copilot, Claude Code, Antigravity
  • Sync GitHub repos: commits, PRs, issues, READMEs
  • AI enrichment: summary + tags + wikilinks per page
  • vis.js knowledge graph with topic clusters and search
  • TF-IDF + LLM /ask with [[citation]] inline answers
  • Plain Markdown — readable by any AI tool
💡
Completely independent

You can run migration against a repo you have never vaulted, and run a vault on a project that has nothing to do with migration. The only thing they share is the terminal and the LLM provider config in ccs.yaml.

Run a migration analysis

shell 
# Analyze a legacy repo and generate the full migration contract

Migration Intelligence — What It Does

The migration engine sits between your source repository and your AI coding agents. It does the expensive, error-prone work that LLMs do badly when given raw code — reading a full codebase, tracing all dependencies, extracting every business rule, and verifying every claim — so your agents can focus on writing correct code rather than reverse-engineering it.

⚠️
The tool never writes code.

CCS prepares the verified context that makes AI rewrites accurate. The contract it produces tells each agent: here is the component, here is what it does, here are its dependencies with source citations, here is the target framework mapping, here are the validation scenarios — now implement it.

Two migration pipelines

/migrate rewrite — Full framework migration

Analyzes an entire codebase for a framework-to-framework rewrite (e.g., .NET → Python, Spring Boot → FastAPI, Express → Go). Discovers every component, extracts behavior, produces a topologically-sorted contract with per-component context docs.

/migrate scan — SOAP / service dependency trace

Traces external SOAP service dependencies in a Node.js repo. Finds every service call, resolves each namespace to its source repo, recursively follows nested calls, and builds a per-service knowledge base with confidence ratings.

Migration Architecture — Three-Tier Engine

Analysis runs in tiers of increasing cost and depth. The LLM is only invoked when static analysis can no longer answer the question.

0
AST & Static Scan Tier 0 — Free
The TypeScript compiler AST parses JS/TS natively. Tree-sitter grammars cover Python, Java, C#, Go, C/C++, Pascal, and VB6. Every import, export, class, function, route, call site, SQL query, and ORM call is extracted without running the code and without calling any LLM. Results are written to code-intelligence.json.
TypeScript compiler API Tree-sitter call graph SQL/ORM extraction zero cost
1
Structure Discovery Tier 1 — Haiku
Key project files (.csproj, pom.xml, Startup.cs, package.json, appsettings.json…) are bundled and sent to Haiku, which identifies the source framework, source language, and recommended target framework. Haiku then groups the file tree into typed components: controller, service, repository, model, dto, middleware, config, utility. Two fast calls, done once per run.
framework detection component grouping 2 LLM calls total
2
Behavior Extraction Tier 2 — Sonnet 4.6
Each component's source files are bundled with a framework mapping table (e.g., ControllerBase → APIRouter, DbContext → SQLAlchemy Session) and sent to Sonnet. The model extracts business rules, data contracts, database interactions, nested service calls, and complexity ratings. It is explicitly instructed to return "unknown" rather than infer where source evidence is absent. Results checkpoint to disk after every component — crashes resume from where they left off.
business rules data contracts framework mapping per-component checkpoint
3
Verification Pass Sonnet 4.6 (configurable)
After extraction, every claim is cross-checked against the source evidence. Claims that cannot be traced to a file and line are flagged. The verifier produces a per-claim audit trail with a trusted, needs_review, or blocked verdict. You can run a different model for verification than for extraction to eliminate shared blind spots between the two passes.
per-claim audit source tracing trust verdict configurable verifier model
4
Topological Sort + Output Generation Free
Components are sorted so leaf nodes (models, repositories) come before their callers (services, controllers). This is the safe implementation order enforced in the migration contract. All artifacts are then written: dashboard HTML, interactive system graph, per-component context docs, Claude Code slash commands, Codex AGENTS.md, and the machine-readable contract JSON.
topological sort dashboard.html vis.js graph agent wiring

⚡
Parser coverage is automatic

Tree-sitter grammars are optionalDependencies and install with npm install. If compilation fails on a restricted machine, CCS falls back to regex silently. The method used per file is recorded in code-intelligence.json.

Migration Outputs

Every run writes a complete set of artifacts to a timestamped run folder. Agent integration files are wired automatically — no manual copy-paste.

Self-contained interactive HTML dashboard — component inventory, complexity heatmap, confidence distribution, trust verdicts. Opens automatically in the browser.
vis.js system graph
Force-directed graph embedded in the dashboard. Nodes = components, target roles, packages. Hover for summary + trust level. Click to drill into the context doc.

Machine-readable contract for agents: target role, implementation gate, evidence ledger, risk flags, validation scenarios, acceptance criteria.
rewrite/context/<Name>.md
Per-component doc: business rules, data contracts, framework mapping table, source links with GitHub line anchors, migration checklist.
verification-summary.md
Per-claim audit trail: evidence, confidence level, source file + line, overall trust verdict.
human-questions.md
Architecture decisions CCS cannot safely infer. Components with unresolved questions are blocked from agent implementation.
component-disposition-matrix.md
Landing-zone map: Azure Function, Databricks, API, workflow, shared lib, adapter, or human review — driven by architecture baseline.
HOW-TO-MIGRATE.md
Step-by-step guide: one step per component, with context doc path, Claude Code command, Codex prompt, and validation scenarios.
.claude/commands/rewrite-<Name>.md
Claude Code slash commands — one per component. Type /project:rewrite-OrderController and the agent has full context.
AGENTS.md
Auto-read by Codex. Migration order, component summaries, context doc locations, target dependency install instructions.
code-intelligence.json
Symbol map, call graph, SQL/ORM extractions, package inventory — queryable via MCP tools.
dependency-risk-report.md
Package inventory with security-sensitive packages flagged and optional OSV advisory matches.

/migrate Commands

Type /m in the terminal UI to see all migrate subcommands with autocomplete.

CommandDescriptionExample
/migrate rewrite Full pipeline: AST scan → detect framework → discover components → extract behavior → verify claims → write all artifacts + dashboard. /migrate rewrite --repo <url> --to csharp --yes
/migrate scan Trace SOAP/WCF external service calls. Resolves namespaces to source repos, follows nested calls recursively. /migrate scan --repo <url> --lang csharp --yes
/migrate reverse-eng Reverse-engineering artifacts only — no migration contract. Good for documentation or pre-planning. /migrate reverse-eng --repo <url> --yes
/migrate dashboard Regenerate dashboard.html for an existing run without re-running the full analysis. /migrate dashboard my-run --open
/migrate status Component-by-component migration progress table with confidence and verification status. /migrate status
/migrate open Open the latest run folder in Finder/Explorer, or open its dashboard. /migrate open --dashboard
/migrate verify Human sign-off on a component context doc. Required before /migrate done. /migrate verify AuthService --by max
/migrate done Mark a verified component as fully rewritten. Requires prior verify. /migrate done AuthService
/migrate db Live read-only database schema extraction. User-approved, credentials never stored. /migrate db --service UserService --yes
/migrate clean Remove old run folders to keep the migration root tidy. /migrate clean --all --yes

migrate rewrite — flags

--repo <url>
GitHub URL of the source repository to analyze. Required.
--to <lang>
Target language: csharp, typescript, python, java, go. Default: python.
--from <framework>
Override auto-detected source framework: aspnet-core, spring-boot, express.
--context <path>
Load a business or architecture doc to anchor LLM decisions. Repeatable. Copied into architecture-context/ for run reproducibility.
--no-context
Disable all context docs, including auto-discovered workspace docs.
--yes
Skip cost confirmation gate. Without this flag the command shows a token estimate and waits for confirmation.

Knowledge Vault — What It Does

The vault is a local, AI-enriched wiki built entirely from your own sources. It does not sync to a cloud service, does not require an account, and never leaves your machine. Every page it creates is plain Markdown — readable by any editor, any AI tool, any agent that can open a file.

pipeline 
GitHub repos  +  IDE histories (Cursor, VS Code, Claude)  +  local files
        │
        ▼ /sync + /harvest + drop into raw/
        │
        ▼ /ingest  →  wiki/ pages (Markdown, auto-merged)
        │
        ▼ /enrich  →  AI summaries + tags + [[wikilinks]]
        │
        ├── /graph   →  interactive vis.js graph in browser
        └── /ask     →  TF-IDF search → LLM answers with citations

Vault Pipeline

1
Vault Init/vault init [path]
Creates the folder structure and saves the vault path to ~/.ccs/config.json. All other commands auto-discover the active vault — you only run this once.
~/.ccs/config.jsonone-time setup
2
Sync/sync
Pulls commits, PRs, issues, and README from GitHub repos listed in ccs.yaml. Writes raw files to raw/github/. Uses GitHub REST API — no local clone needed.
GitHub REST APIraw/github/GH_TOKEN
3
Harvest/harvest
Extreme Scraper logic for the AI era. Mines conversation history from Cursor generations and composer histories, VS Code Copilot .jsonl sessions, Claude Code project logs, and Antigravity brain memories. Writes one file per memory to raw/memories/ — no manual export needed.
CursorVS Code CopilotClaude CodeAntigravityraw/memories/
4
Ingest/ingest
Reads everything in raw/ and converts it to Markdown wiki pages in wiki/. Re-running merges new content — never overwrites existing pages. HTML → wiki/concepts/, conversations.json → wiki/conversations/. Every page gets standard YAML frontmatter.
HTML parserMarkdown mergeYAML frontmatter
5
Enrich/enrich
Sends each un-enriched wiki page to the configured LLM. Gets back a one-paragraph summary, 3–6 keyword tags, and related page titles. Injects summary and tags into frontmatter. Appends ## Related section with [[wikilinks]]. Safe to re-run — skips already-enriched pages. Idempotent.
LLM per-page[[wikilinks]]frontmatter injectionidempotent
6
Graph/graph
Builds a self-contained interactive vis.js knowledge graph from wiki/ and opens it in the browser. One node per wiki page. Size scales with connection count — hubs are visually prominent. Color groups pages by topic: code=purple, AI=green, design=pink, data=yellow, devops=blue. Glow shadow per color (Obsidian-style). Hover fades unconnected nodes, brightens the connected cluster. Click opens the summary panel. Search bar filters live. Legend click isolates a topic group.
vis.jsTF-IDF edgesObsidian glowoutput/graph.html
7
Ask/ask <question>
TF-IDF keyword search across all wiki pages. Top 6 matching pages injected into LLM context. LLM answers with [[page-title]] inline citations. Title token matches score 5×, tag matches 3×, body term frequency 1 + log(freq), normalized by query length.
TF-IDF retrievalRAG[[citations]]top-6 context

Vault Structure

The vault is plain Markdown — any AI tool that reads files can use it directly.

vault/ ├── ccs.yaml ← sources + LLM provider config ├── raw/ ← sync inbox (never edit manually) │ ├── github/ ← /sync writes here │ ├── memories/ ← /harvest writes here │ └── uploads/ ← drop local files here ├── wiki/ ← auto-maintained by /ingest + /enrich │ ├── _master-index.md ← /index rebuilds this │ ├── concepts/ ← from HTML + text │ └── conversations/ ← from conversations.json ├── output/ │ ├── graph.html ← /graph output │ └── rewrite-context/ ├── skills/ │ ├── wiki-ingest/SKILL.md │ ├── wiki-query/SKILL.md │ ├── graph-build/SKILL.md │ └── rewrite-plan/SKILL.md └── .claude-plugin/plugin.json ← Claude Code manifest ~/.ccs/ ├── config.json ← global: stores activeVault path └── guide.html ← /guide output

/enrich, /graph, /ask — key behaviors

/enrich — idempotent
Skips pages that already have a summary: field. Safe to re-run at any time — only un-enriched pages are processed.
/graph — edge building
TF-IDF keywords from title + summary. Keywords appearing in 2–30 nodes create edges. Max 12 connections per node. Keywords in >30 nodes are too common to be meaningful edges.
/ask — scoring
Title token match: 5× · Tag match: 3× · Body frequency: 1 + log(freq) · Normalized by query length. Top 6 pages injected into LLM context.
/index
Rebuilds wiki/_master-index.md — a single file listing every wiki page with title, type, and tags. Useful for quick grep or AI context injection.

Configuration — ccs.yaml

Every vault has a ccs.yaml at its root. This file controls GitHub sources and the LLM provider used by /enrich, /ask, and /migrate.

yaml 
sources:
  - type: github
    repos:
      - my-org/auth-svc
      - my-org/payment-svc
    include: [commits, prs, issues, readme]
    token_env: GH_TOKEN

llm:
  provider: anthropic        # anthropic | openai | enterprise
  model: claude-sonnet-4-6

  # Optional: use a different model for the verification pass
  verifier_provider: anthropic
  verifier_model_flash: claude-haiku-4-5-20251001

LLM Providers

Anthropic
ANTHROPIC_API_KEY
Claude Haiku + Sonnet 4.6. Default provider. Best balance of speed and quality for migration analysis.
OpenAI
OPENAI_API_KEY
GPT-4o and family. Use as primary or as independent verifier to reduce shared blind spots.
Enterprise
CCS_ENTERPRISE_*
Azure OpenAI with OAuth2 client credentials. Set CCS_ENTERPRISE_CLIENT_ID, CLIENT_SECRET, AUTH_URL, SCOPE, API_BASE.

AI Tools & MCP Integration

CCS exposes a local MCP server that lets your coding agent read all migration artifacts directly. Start it with ccs-code mcp.

shell 
# Start the MCP server
ccs-code mcp

# Register with Codex CLI
codex mcp add ccs -- ccs-code mcp

# Add to Claude Code .mcp.json
json 
{
  "mcpServers": {
    "ccs": {
      "command": "ccs-code",
      "args": ["mcp"]
    }
  }
}
